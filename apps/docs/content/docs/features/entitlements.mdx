# Entitlements

Entitlements are the core of Owostack's permission system. They define **who** can do **what** based on their subscription plan.

## Concepts

### Features

A **Feature** represents a capability in your application. It is the atomic unit of access control.

| Type      | Description                         | Example             |
| --------- | ----------------------------------- | ------------------- |
| `boolean` | Binary access (on/off)              | `can_access_api`    |
| `metered` | Numeric limit with usage tracking   | `api_calls_monthly` |
| `static`  | A fixed value (e.g., configuration) | `custom_domain`     |

### Plans

A **Plan** is a subscription tier (e.g., "Basic", "Pro", "Enterprise"). A plan groups a set of features with specific configurations.

Example Plan Configuration:

- **Pro Plan**
  - `can_access_api`: ✅ Allowed
  - `api_calls_monthly`: Limit 10,000 / month
  - `custom_domain`: ✅ Allowed

## Checking Access

You check entitlements using the `check()` method in the SDK. This method automatically resolves the customer's active subscription, looks up the plan, and verifies the feature configuration.

```ts
const { allowed, code, remaining } = await owo.check({
  customer: "user@example.com",
  feature: "api_calls_monthly",
});

if (allowed) {
  // Proceed with request
} else {
  // Handle denial (e.g., upgrade prompt)
}
```

### Response Codes

- `access_granted`: User has access.
- `limit_exceeded`: User has hit their usage limit.
- `feature_not_in_plan`: The feature is not available on the current plan.
- `no_active_subscription`: User has no active subscription.

## Overrides

Owostack allows you to override
entitlements at the **Customer** level. You can grant specific features to a customer regardless of their plan (e.g., "Grandfathering" early users or giving beta access).
