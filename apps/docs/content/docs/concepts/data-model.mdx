# Data model

Owostack stores a small set of objects that let it answer two questions quickly:

- Is the customer allowed to use feature X right now?
- If they use feature X, how do we account for it (meters/credits)?

## Core objects

- Organization
- Project
- Customer
- Plan
- Feature
- Subscription
- Entitlement
- Usage / Meter state
- Transactions / events

## Provider references

Most billing objects store provider IDs (for example, a provider subscription code) so Owostack can reconcile internal state with provider state.

## Environments

Use separate environments for test and live credentials so you can validate integration safely.
