# Catalog & Sync Model

Owostack supports two ways to manage features and plans: through the **dashboard** (point-and-click) and through **code** (the SDK catalog). This page explains how the two coexist and what happens when they overlap.

## Two sources of truth

Every feature and plan in Owostack has a `source` field:

- **`"dashboard"`** — created or last modified through the dashboard UI
- **`"sdk"`** — created or last modified through `owo.sync()`

This distinction drives conflict resolution and determines what sync touches.

## What sync does

When you call `owo.sync()` (or `npx owostack sync`), the SDK:

1. **Extracts** unique features from all plans in your catalog
2. **Sends** the full catalog to `POST /api/sync`
3. **Server reconciles** each feature and plan:
   - If it doesn't exist → **create** it (tagged `source: "sdk"`)
   - If it exists and properties changed → **update** it (tagged `source: "sdk"`)
   - If it exists and nothing changed → **leave it** (mark as `source: "sdk"` if not already)
4. **Reconciles plan features** — adds/updates/removes the feature configs attached to each plan
5. **Returns** a change report

## What sync does NOT do

- **Never deletes** features or plans. Removing a feature from your catalog code does not delete it from the API. Use the dashboard to archive or delete resources.
- **Never touches dashboard-only resources.** If you created a plan in the dashboard and it's not in your catalog, sync ignores it entirely.
- **Never syncs with payment providers.** Sync creates features and plans in the Owostack database. Provider sync (creating a Paystack plan, for example) happens separately when a customer checks out.

## Conflict resolution

Sync follows a **code-wins** policy for resources it manages:

| Scenario | What happens |
|----------|-------------|
| Feature exists only in code | Created on sync |
| Feature exists only in dashboard | Untouched |
| Feature exists in both, code changed | Code version wins, resource tagged `source: "sdk"` |
| Feature exists in both, dashboard changed after last sync | Next sync overwrites with code version |
| Feature removed from code but exists in API | Nothing — resource stays |

The dashboard shows a badge for SDK-managed resources: **"Managed by SDK"**. Editing an SDK-managed resource in the dashboard triggers a warning that it will be overwritten on next sync.

## Recommended workflow

### Small teams / early stage

Use the catalog for everything. Define features and plans in code, sync on deploy. Dashboard is read-only for verification.

```
Code → sync → API → Dashboard (read-only)
```

### Larger teams / product managers involved

Use the catalog for core features and plans. Let product managers create experimental plans in the dashboard. The two don't interfere.

```
Code → sync → core features/plans
Dashboard → experimental features/plans
```

### Migration from dashboard to code

If you've been using the dashboard and want to move to code:

1. Define your existing features and plans in a catalog
2. Run `owo.sync()` — it will match by slug and tag them as `source: "sdk"`
3. Future changes go through code

No data is lost. Existing subscriptions, usage records, and entitlements are unaffected.

## Feature handles vs. sync

The catalog serves two purposes:

1. **Feature handles** — `metered()` and `boolean()` return handles you can use for `.check()` and `.track()`. These work immediately, with or without syncing.
2. **Sync** — pushing the catalog to the API creates the features and plans in the database.

You can use feature handles without ever calling sync. The handles are just convenience wrappers around `owo.check({ feature: "slug" })`. Sync is only needed when you want the API to know about your plans and features (for the dashboard, for `attach()` checkout flows, etc.).
